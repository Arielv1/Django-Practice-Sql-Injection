What is SQL Injection

SQL Injection, also known as SQLi, is a common attack vector that uses malicious SQL code execution for backend database manipulation.

What are SQL queries

SQL is a standardized language used to access and manipulate databases to build customizable data views for each user. SQL queries are used to execute commands, such as data retrieval, updates, and record removal. Different SQL elements implement these tasks, e.g. , queries using the SELECT statement to retrieve data, based on user-provided parameters.

SELECT ItemName, ItemDescription
FROM Items
WHERE ItemNumber = ItemNumber

The commands are different for each database for example: Using postgre database you will need to do this command to get the name of the database:Select current_database()

Using MySQL database will be this command: Select database()

For more details regarding general SQL commands see references at the bottom of the page.
See also: SQL Commands Cheat Sheet

Types of SQL Injection

SQL Injections typically fall under three categories: In-band SQLi (Classic), Blind SQL injection and Out-of-band SQL injection.
You can classify SQL injections types based on the methods they use to access backend.

For useful SQL Injection commands for various databases see: SQL Injection Cheat Sheet

SQL Injection prevention and mitigation

1: Use of Prepared Statements (with Parameterized Queries).
2: Use of Stored Procedures.
3: Whitelist Input Validation.
4: Escaping All User-Supplied Input.
5: Using external applications.

Prepared Statements

These are SQL statements that are sent to and parsed by the database server separately from any parameters.

The Query sent to server before the data input sent to him, and only when the execute happens the data input will be added up to the query. Which means that attacker cant alter the database. This methods cannot used on Dynamic queries where the query using input on field names or dynamic table names. and sometimes can be slower compare to other approaches.

This methods is regarded as best practice, it's recommended you use it whenever possible

Stored Procedures

The difference between prepared statements and stored procedures is that the SQL code for a stored procedure is defined and stored in the database itself, and then called from the application.
Stored Procedures should also check which privileges the users are using, on certain scenarios a user can have admin privileges which is best avoided.

Whitelist Input Validation

A technique which limits the options of the users and as a result limits the possible potential of SQL injections.
Whitelisting is used in this situation to check the users' input from a pre-defined allowed input list - for example checking which item was selected from a list.

Escaping All User-Supplied Input

A method that alters and modifies the users' input in order to 'escape' from potentially dangerous key codes, label and variables such as \ ' or *

Least Privilege

A defensive technique in which the users are grouped and are given privileges which SQL commends that can perform based upon in which group they're part of.

Using external applications

Just to let you dear readers know that there are more options, like Web application firewalls.
