What is SQL Injection

SQL Injection, also known as SQLI, is a common attack vector that uses malicious SQL code execution for backend database manipulation.

What are SQL queries

SQL is a standardized language used to access and manipulate databases to build customizable data views for each user. SQL queries are used to execute commands, such as data retrieval, updates, and record removal. Different SQL elements implement these tasks, e.g. , queries using the SELECT statement to retrieve data, based on user-provided parameters.

SELECT ItemName, ItemDescription
FROM Item
WHERE ItemNumber = ItemNumber

The commands are different for each database for example: Using postgre database you will need to do this command to get the name of the database:Select current_database()

Using MySQL database will be this command: Select database()

For more details regarding general SQL commands see references at the bottom of the page.
See also: SQL Commands Cheat Sheet

Types of SQL Injection

SQL Injections typically fall under three categories: In-band SQLi (Classic), Blind SQL injection and Out-of-band SQL injection.
You can classify SQL injections types based on the methods they use to access backend.

For useful SQL Injection commands for various databases see: SQL Injection Cheat Sheet

3. Out-band SQL injection

The attacker uses the different channel of communication to launch their attacks and to gather their results.
In order to perform this sort of injections the attack uses third party software in order to listen to the communication to the database and modify the packets as they are being sent.
One of the tools that are used for this type of injection is Burpsuite

SQL Injection prevention and mitigation

Option 1: Use of Prepared Statements (with Parameterized Queries)
Option 2: Use of Stored Procedures
Option 3: Whitelist Input Validation
Option 4: Escaping All User-Supplied Input
Option 5: Using external applications.

Prepared Statements

These are SQL statements that are sent to and parsed by the database server separately from any parameters.

The Query sent to server before the data input. and only when the execute happens it add up to the query. Which means that attacker cant alter the database. This methods cannot used on Dynamic queries where the query using input on field names or dynamic table names. and sometimes can be slower compare to other approaches.

Stored Procedures

The difference between prepared statements and stored procedures is that the SQL code for a stored procedure is defined and stored in the database itself, and then called from the application Stored Procedures also should be check which privileges they using sometimes it can be admin rule which you prefer to avoid it..

Whitelist Input Validation

Check the user input on specific list of options that should be valid. This option is good on Dynamic Query when Prepared Statement cant be used.

Escaping All User-Supplied Input

This technique should only be used as a last resort, This method is to escape special characters so the sql statement will not get alter by attacker using special character like. ' or ".
